# Complete UI Audit — April 2, 2026

## CRITICAL WORKFLOW BUGS (Blocking daily operations)

### WORKFLOW-1: "Create Ticket" creates an invoice — SHOULD NOT
- **Where:** POS > Create Ticket flow. After adding items to cart, clicking "Create Ticket" shows "Sale Complete! Invoice #858" — but no sale happened. Customer just dropped off a device.
- **Impact:** Invoice is created with $0 paid. When tech later tries to checkout the same ticket, it fails with "UNIQUE constraint failed: invoices.ticket_id" because an invoice already exists.
- **Root cause:** `checkout-with-ticket` endpoint uses `mode: 'create_ticket'` but still creates an invoice in both modes. The success screen says "Sale Complete" even for ticket-only creation.
- **Fix:**
  1. When `mode === 'create_ticket'`: DO NOT create invoice. Only create ticket + devices + parts.
  2. Change success screen text from "Sale Complete!" to "Ticket Created!" when mode is create_ticket
  3. Remove "Invoice #XXX" from success screen for ticket-only creation
  4. When `mode === 'checkout'` with `existing_ticket_id`: check if invoice already exists for this ticket. If so, use the existing invoice instead of creating a new one.

### WORKFLOW-2: Wrong repair service name attached to ticket
- **Where:** POS > Mobile > iPhone 15 > Screen Replacement — ticket shows "OLED Assembly for iPhone 17 Pro Max (Aftermarket Soft / QV8 / 120HZ)"
- **Impact:** Receipt and ticket show completely wrong part name. Customer sees iPhone 17 Pro Max part on their iPhone 15 repair.
- **Root cause:** When no preset repair price exists, the system still attaches a `repairServiceId` from the repair pricing table that doesn't match the selected device. The ID is from a different device's service.
- **Fix:** When `repairServiceId` is null or doesn't match the device, use the service NAME (e.g., "Screen Replacement") as the description, not a random catalog part. The service_name from the drill-down should be stored on ticket_devices.service_name.

### WORKFLOW-3: Checkout for existing ticket fails (500 error)
- **Where:** Ticket detail > Checkout button > POS > Complete Checkout
- **Impact:** Checkout fails silently — modal stays open, no error shown to user. Server returns 500: "UNIQUE constraint failed: invoices.ticket_id"
- **Root cause:** Create Ticket already made an invoice. Checkout tries to make another one for the same ticket.
- **Fix:** Same as WORKFLOW-1 fix #4 — checkout should detect existing invoice and use it, OR don't create invoice during ticket creation.

### WORKFLOW-4: No error feedback on checkout failure
- **Where:** Checkout modal — when 500 occurs, nothing happens. Button looks clickable, no toast/error.
- **Fix:** Add error handling in `handleCompleteCheckout` — show toast.error with the server message.

## CRITICAL FLOW REDESIGN NEEDED

### FLOW-1: Separate "Create Ticket" from "Checkout" clearly
The current flow conflates two different operations:
- **Drop-off flow:** Customer drops off device → search customer → pick device → fill details → **Create Ticket** (NO payment, NO invoice)
- **Pickup/Checkout flow:** Open existing ticket → verify work done → **Checkout** (create invoice + record payment + close ticket)

**Currently:** Both flows go through the same POS endpoint and both create invoices. The success screen says "Sale Complete" for both.

**Proposed redesign:**
1. "Create Ticket" button → calls `mode: 'create_ticket'` → creates ticket only → shows "Ticket Created! T-XXXX" with Print Label / View Ticket / New Check-in buttons
2. "Checkout" button → calls `mode: 'checkout'` → creates invoice + payment + closes ticket → shows "Payment Received! $XXX" with Print Receipt / View Invoice / New Sale buttons

### FLOW-2: Parts should show as separate line items on receipt/invoice
- **Current:** Cart shows "Apple iPh... $149.99" — one line for the entire repair
- **Needed:** When parts are attached to a repair, the receipt should show:
  - Line 1: "iPhone 15 Screen Replacement (labor)" — $80.00 — No tax
  - Line 2: "iPhone 15 OLED Screen Assembly (part)" — $69.99 — Tax: $6.21
  - This way the customer sees exactly what they're paying for
- **Where to fix:** Cart display in LeftPanel.tsx already shows parts under the repair. But on the receipt/invoice, parts need to be separate line items with their own tax calculation.

### FLOW-3: Service selection should show prices
- **Where:** POS > Select Service screen — "Screen Replacement", "Battery Replacement" etc. are plain text pills with no price
- **Fix:** Show price next to service name when available: "Screen Replacement — $149.99", "Battery Replacement — $79.99". When no price configured, show "Screen Replacement — Custom price"

### FLOW-4: Post-add-to-cart flow is unclear
- **Where:** After adding an item to cart, the right panel resets to category selection with no guidance
- **Fix:** Show a prominent banner: "Added to cart! Add another device or click Create Ticket when ready." Or auto-focus the bottom action buttons.

### FLOW-5: Cart item names are truncated
- **Where:** Left panel cart shows "Apple iPh..." — not enough info to distinguish items
- **Fix:** Show at minimum "iPhone 15 - Screen Repl." or use two lines: device name on top, service on bottom

## CRITICAL BUGS (Broken functionality)

### BUG-1: POS Checkout doesn't record payment or close ticket for existing tickets
- **Where:** Ticket T-2904 shows "Paid: $0.00", "Due: $34,001.81" after checkout, status still shows "Waiting for inspection" on devices
- **Root cause:** The `existing_ticket_id` flow was just added but the server may be creating a new ticket instead of reusing the existing one when coming from the POS ticket list. The ticket status on the header says "Repaired" but devices say "Waiting for inspection" — device statuses are not synced with ticket status.
- **Fix:** Debug the checkout-with-ticket flow end-to-end. Verify `existing_ticket_id` is being sent correctly. When ticket status changes, also update all device statuses to match. Payment must be recorded against the invoice.

### BUG-2: Ticket detail — device statuses don't match ticket status
- **Where:** Ticket T-2904 header shows "Repaired" status but each device card shows "Waiting for inspection"
- **Fix:** When ticket status changes, optionally sync device statuses. Or at minimum, make the device status independently changeable from the device card.

### BUG-3: POS cart persists stale data from previous session
- **Where:** POS page shows 2 items from a previous checkout session (Test Item 1, Test Item 2) even though those tickets were already created
- **Fix:** After successful ticket creation or checkout, ensure `resetAll()` is called AND the localStorage is cleared. The persist middleware may be saving state before resetAll completes.
