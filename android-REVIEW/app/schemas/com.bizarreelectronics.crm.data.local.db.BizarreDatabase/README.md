# Room schema history

This directory is auto-generated by the Room compiler (KSP) when
`exportSchema = true` on the `@Database` annotation. Each file records the
full schema — table DDL, fields, indices, foreign keys, and an identity hash
— at one database version.

## AUD-20260414-L1: 3.json is intentionally missing

`1.json`, `2.json`, and `4.json` are present. **`3.json` is not**, even though
`MIGRATION_2_3` and `MIGRATION_3_4` both exist in
`data/local/db/Migrations.kt`. Version 3 was a real shipped-internally
schema version — it was the `@Database(version = ...)` value on the commit
that introduced `MIGRATION_2_3` — but the matching JSON was not committed to
git at the time, and the next audit pass bumped the version to 4 without
back-filling it (see the comment block in `BizarreDatabase.kt`).

### Why 3.json was not reconstructed

Rebuilding the v3 schema JSON requires running the Room compiler against the
v3 entity classes, because Room's `identityHash` is the MD5 of the entities'
generated `CREATE TABLE` strings — hand-writing a JSON file with the wrong
hash would pass human review but be silently rejected by
`MigrationTestHelper`. The v3 code lives in a prior commit, but parts of the
app (navigation shortcuts, widget, shared brand components, etc.) have moved
forward since then, so checking out that commit in-place does not build
cleanly without a large, unrelated revert. For a Low-priority audit finding,
that surgery is not worth the risk.

### What we do instead

1. `Migrations.MIGRATION_3_4` is fully documented in `Migrations.kt` — the
   per-column, per-index diff from v3 to v4 is readable there.
2. `RoomSchemaFilesTest` (under `app/src/test/...`) asserts that the
   currently-shipped schema files remain present, so future schema bumps
   cannot silently delete 1/2/4.json the way 3.json was lost.
3. If a future engineer needs the exact v3 schema (for a migration-test
   harness or forensic reason), they can:
   - Check out the commit where `BizarreDatabase.kt` last had `version = 3`
     into a throwaway worktree.
   - Run `./gradlew :app:kspDebugKotlin` in that worktree to regenerate
     3.json.
   - Copy the file into this directory on the main branch.
   - Delete `documentedGapForVersion3Still` from `RoomSchemaFilesTest`.
   - Add `"3.json"` to `REQUIRED_SCHEMAS`.

## Adding a new schema version

When you bump `@Database(version = N)`:

1. Add `MIGRATION_{N-1}_{N}` to `Migrations.kt` and register it in
   `ALL_MIGRATIONS`.
2. Build — KSP writes `N.json` here.
3. Commit `N.json` alongside the entity/migration changes. Do not `.gitignore`
   it.
4. Add `"N.json"` to `RoomSchemaFilesTest.REQUIRED_SCHEMAS`.
