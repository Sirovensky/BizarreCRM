## Thoroughness bar — non-negotiable

This is a high-stakes review. Shallow scans are unacceptable. Plan to spend **AT LEAST 25–45 minutes and 60+ tool calls** of investigation per slot. Do NOT return early. If you finish a quick pass and feel "done", you are NOT done — dig deeper.

Required investigation steps:

1. **Read every focus file end-to-end.** No skimming. For files over ~800 lines, read them in ranges; cover the entire file.
2. **Follow every relevant import.** Read the helpers, validators, middleware, DB layer the focus files call into. Security bugs hide in the helpers, not the entry points.
3. **Grep exhaustively.** Use multiple keyword variants, regex patterns, and synonyms. Don't rely on a single sample.
4. **Trace data flow end-to-end.** Where does each user-controlled input originate? What mutations happen in body parsing, validation, middleware? Where does it land — SQL? Disk? Network? Logs? Email? SMS?
5. **Verify each suspicion before recording it.** Read callers, middleware ordering in `index.ts`, DB schema in `db/migrations/`, and actual constants. A finding without a verified line citation is rejected.
6. **Cross-check framework behavior.** better-sqlite3 escaping, jsonwebtoken alg pinning, Express middleware order semantics, ws upgrade authz, Node `crypto` modes — don't assume; verify by reading the call site against documented behavior.
7. **Hunt OWASP + business-logic + composition.** OWASP Top 10 (auth, injection, IDOR, SSRF, XXE, deserialization, logging, etc.), business-logic flaws (TOCTOU, race on counters/balances, off-by-one in money/dates, replay, idempotency gaps, sign-confusion in refunds/credits), composition flaws (middleware order, async req-scope leak, tenant context drift across cron/WS, missing audit on privileged ops).
8. **Negative-space search — what's missing?** Missing rate limit, missing role gate, missing tenant filter, missing audit log, missing CSRF, missing escape, missing input length cap, missing DB transaction, missing constant-time compare, missing token expiry.
9. **Read related migrations.** A bug is often only visible when you compare the route's WHERE clause to the table's actual columns/indexes/constraints in `packages/server/src/db/migrations/`.
10. **Don't stop at the first 2–3 findings.** Many slots have 5–10+ real issues. Push until you've genuinely exhausted the scope.

Append findings to your assigned file. Use this exact structure per finding (Markdown):

```
**Code:**

\`\`\`

**Exploit:**
[1–3 sentences: concrete attacker scenario, including the request/payload they send and the impact.]

**Fix:**
[1–3 sentences: specific remediation. Mention the exact API/check/constant to add or change.]

---
```

`SEV` is one of: **CRITICAL**, **HIGH**, **MEDIUM**, **LOW**, **INFO**.

Severity guide:
- **CRITICAL** — unauthenticated RCE, full cross-tenant data exposure, master credential bypass, money manipulation by anyone.
- **HIGH** — authenticated cross-tenant or privilege escalation, signature bypass on payments/webhooks, account takeover via reset/2FA flaw, secret exposure to wrong audience.
- **MEDIUM** — IDOR within tenant, missing rate limit on sensitive flow, predictable token, exposed PII, weak crypto with limited blast radius, CSRF gap on state-changing route.
- **LOW** — information leak (timing/enumeration), weak default with mitigations, missing defense-in-depth.
- **INFO** — hardening recommendation, observation worth tracking.
